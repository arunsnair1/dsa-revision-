# DSA spaced repetition tracker — v2 fix verification (v2)

All seven issues from the v1 review have been addressed. The fixes are well-scoped: import validation is rigorous, the struggled-feedback penalty is no longer asymmetric, carry-forward prevents deferred-item pile-up, localStorage quota failures notify the user, the deadline auto-advances past its hardcoded date, UTC date drift is eliminated, and demoted items are visually distinct.

**Watch for:** The `carryForwardDeferred` method runs unconditionally on every page load and saves immediately, meaning a user who opens multiple tabs could stomp each other's state at a day boundary (possible). No blocking issues remain.

**Verdict**: APPROVED

## High-level view

The carry-forward mechanism bumps deferred items to tomorrow at startup. Items already pushed to tomorrow don't match the `nextReviewDate <= today` filter on a second load, making the operation effectively idempotent within a single calendar day. The multi-tab concern is limited to the midnight-boundary edge case (Tab A at 11:59pm, Tab B at 12:01am).

Import validation now performs `Array.isArray`, per-entry field type assertions, and fills missing optional fields. Fields like `successCount` and `type` are intentionally not validated — they self-heal on the next feedback cycle via defaults.

The local-date helpers (`getToday`, `addDays`, `getDaysDiff`) parse date strings into local-midnight `Date` objects using component parts rather than the `Date(string)` constructor, avoiding both UTC drift and browser-dependent date parsing.

The `getDefaultDeadline` function computes 30 days via millisecond arithmetic when the hardcoded deadline has passed — this could land on a different calendar day near DST transitions (at most a one-day discrepancy).

## Carry-forward idempotency and the `deferred` flag

The method filters for `nextReviewDate <= today`, sorts by overdue count and struggled-first, then packs items against the budget. Overflow items get `nextReviewDate = tomorrow`. Because the save happens immediately, a second tab on the same day loads already-bumped state and the filter excludes those items — effectively idempotent.

The edge case: Tab A at 11:59pm sets tomorrow to the date that Tab B (opened at 12:01am) considers today. Tab B re-processes those items and may bump them again to the *next* day, causing a one-day gap in their schedule.

The `deferred` flag (`item.scheduleRef.deferred = true`) persists in localStorage but is never read or cleared anywhere. It was likely intended for future UI differentiation but currently accumulates as dead storage. Consider either removing the write or adding a query point that uses it.
