# DSA spaced repetition tracker — static SPA for Neetcode 150 scheduling

A single-page application for scheduling DSA problem reviews using spaced repetition. The four-file static site (no build step, no dependencies) implements a full revision workflow: mark questions complete, receive daily review schedules bounded by a time budget, provide feedback that adjusts future intervals, and track mastery progression across 150 problems. Persistence is localStorage-only with JSON export/import for portability.

**Watch for:** Deferred items silently vanish until the next page load with no carry-forward mechanism (confirmed). The `'struggled'` feedback path can trap items at interval index 0 indefinitely while still decrementing `successCount` below the threshold it guards (confirmed). Import validation is shallow enough that malformed `revisionSchedule` entries will crash rendering (confirmed).

## High-level view

The spaced repetition engine maps three feedback levels onto a five-step Ebbinghaus interval ladder [1, 3, 7, 14, 30 days]. The `successCount` decrement on struggle creates a dead-end where mastery requires far more reviews to achieve after a single regression than the straight-through path — the penalty is asymmetrically harsh relative to recovery.

The scheduler enforces the 180-minute daily budget by sorting due items (overdue-first, then struggled, then by category) and packing them greedily. Items that overflow the budget are labeled "deferred" but never rescheduled — they reappear as overdue tomorrow, indistinguishable from genuinely overdue reviews. A demotion heuristic converts overflow re-solve items to read-notes to squeeze them in, but this creates a mismatch between displayed time allocation (5 min) and actual cognitive demand (20 min re-solve).

Import validation checks only for top-level property presence (`completedQuestions`, `revisionSchedule`), so corrupt inner records pass validation but crash rendering downstream. All date logic uses UTC rather than local time, which shifts "today" for users in negative UTC offsets studying in the evening.

1. **Deferred items have no carry-forward** — Items that exceed the daily budget reappear as overdue the next day with increasing overdue counts, but there is no distinct treatment for "deferred yesterday" vs "genuinely overdue". The overdue-first sort guarantees deferred items keep blocking new reviews until the budget is raised. Consider bumping their `nextReviewDate` forward by one day on deferral.
2. **Struggled feedback traps items below mastery** — `successCount` decrements on `struggled` and mastery requires `successCount >= 4`. A user who struggles once after three good reviews drops to count 2 *and* interval index regresses, requiring effectively five more good reviews to reach mastery. Consider not decrementing below zero or decoupling count from interval regression.
3. **Import validates structure but not shape** — `importData` accepts any truthy values for `completedQuestions` and `revisionSchedule`. A non-array `revisionSchedule` (e.g., a string) will throw on `.filter()` in `getDailyItems`. Schedule entries missing `questionId` will surface as `undefined.intervalIndex` in `getMasteryLevel` via `ProgressTracker.getStats`. Add `Array.isArray` and per-entry field checks.
4. **localStorage quota has no user feedback** — `save()` catches exceptions silently. On quota-exceeded, the user continues with in-memory state that vanishes on reload. Surface a notification prompting data export.
5. **Hardcoded deadline will rot** — `DEADLINE = "2025-06-25"` causes a confusing negative countdown for any new user after that date. Default to 30 days from first load instead.
6. **Demotion heuristic misleads on time** — When a re-solve item is demoted to read-notes to fit the budget, the dashboard shows "Read Notes (5 min)" but the item still requires full re-solve cognitive effort. The user's actual time investment won't match the budget display.
7. **UTC date logic shifts "today" for evening studiers** — `new Date().toISOString().split('T')[0]` returns UTC date. Users in UTC-8 studying after 4pm see tomorrow's date as today, causing reviews to not appear when expected and activity heatmap entries to land on the wrong day.

## Interval ladder and mastery progression

The core scheduling logic lives in `SpacedRepetitionEngine.scheduleRevision`. On first completion, an item enters at interval index 0 (review in 1 day) with `successCount: 0`. Each subsequent review shifts the index based on feedback:

```javascript
if (feedback === 'easy') nextIndex = currentIntervalIndex + 2;
else if (feedback === 'struggled') nextIndex = Math.max(0, currentIntervalIndex - 1);
else nextIndex = currentIntervalIndex + 1;
```

The `successCount` gating for mastery (`intervalIndex >= 4 && successCount >= 4`) means an item must reach the 30-day interval *and* accumulate four net successes. Because struggled decrements `successCount`, an item at index 3 with `successCount: 3` that gets one "struggled" drops to index 2 and `successCount: 2`. It now needs to climb back to index 4 (three good reviews minimum) while also recovering the count — effectively five more good reviews. The asymmetry between the penalty (drop one index, lose one count) and the recovery (one good review gains one index and one count) means the struggled-then-recovered path is disproportionately longer. Decoupling `successCount` from the interval regression would make the system more forgiving without sacrificing the conservative intent.

## Budget overflow and demotion heuristic

```javascript
if (item.type === 're-solve' && totalTime + TIME_READ_NOTES <= budget) {
  item.type = 'read-notes';
  item.timeEstimate = TIME_READ_NOTES;
  scheduled.push(item);
}
```

The demotion mutates a spread copy, not the original schedule entry, so state isn't corrupted across renders. But the user-facing gap remains: the dashboard displays a demoted item under "Read Notes (5 min each)" while the underlying problem demands 20-minute re-solve effort. The feedback buttons still work correctly (they look up the original schedule entry), but the time budget bar underreports actual study time needed.

Items that don't fit even after demotion are shown as "Deferred to Tomorrow" but their `nextReviewDate` is not updated. Tomorrow they appear with `daysOverdue: 1`, sorted to the top, potentially displacing items that are genuinely overdue from earlier missed sessions. Over time, a persistent budget deficit creates an ever-growing overdue queue that monopolizes the schedule.

## Import/export trust boundary

```javascript
if (data.completedQuestions && data.revisionSchedule) {
  return data;
}
```

This passes if `completedQuestions` is `[]` (truthy) and `revisionSchedule` is `"hello"` (truthy). Downstream, `Scheduler.getDailyItems` calls `.filter()` on `state.revisionSchedule` — which throws on a string. More subtly, `ProgressTracker.getStats` iterates `QUESTIONS` and calls `.find()` on `revisionSchedule` for each, then passes the result to `getMasteryLevel`. A corrupt entry like `{questionId: 999}` is caught by the dashboard's `.filter(r => r.question)` guard, but `getStats` has no such guard — `getMasteryLevel` receives a schedule entry and accesses `.intervalIndex` and `.successCount`, which would be `undefined` for a malformed record rather than crashing, but would produce incorrect mastery classifications.

A lightweight schema check on import (verify `Array.isArray(data.revisionSchedule)`, verify each entry has numeric `questionId` and `intervalIndex`) would prevent both crash and silent corruption paths.

## Unbounded history growth and localStorage durability

The history array grows by one entry per feedback action and per "mark complete". Over months of daily use (150 questions, multiple review cycles each), this reaches thousands of entries. The settings view renders only the last 50, but the full array persists in localStorage. Combined with the silent-failure `save()`, a user could hit quota limits without warning and lose all subsequent state on reload.

A periodic trim (keep last 500 entries or entries from the last 90 days) would bound storage growth. Surfacing a non-blocking error when `localStorage.setItem` throws would prevent silent data loss.

## Date handling and timezone sensitivity

All date logic uses `new Date().toISOString().split('T')[0]`, which returns the UTC date. A user in UTC-8 who studies at 10pm local time gets tomorrow's UTC date as "today". This means reviews scheduled for "today" might not appear until the user's morning (when UTC catches up to their local calendar date), and items completed at night get recorded as the next day, skewing the activity heatmap. Switching to `toLocaleDateString('en-CA')` (returns YYYY-MM-DD in local time) would align the app's notion of "today" with the user's wall clock.
